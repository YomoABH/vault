Анализ обёртки IndexedDB: миграции и версии
Context
Текущая обёртка src/infrastructure/persistence/indexeddb/index.ts была достаточной для Phase 1: один store notes, без индексов, без эволюции схемы. Phase 2.1 (Folders) — первая точка, где эта простота ломается: нужно добавить store folders, индекс folderId на notes, и backfill существующих записей (folderId: null). Это первая реальная миграция v1 → v2 в проекте, и она обнажает три ограничения, которые ты сам назвал.

Задача этого документа — не написать код, а дать карту: что именно не так, какие концепции IndexedDB нужно изучить, и какие архитектурные развилки у тебя есть. Плана имплементации здесь нет намеренно — пишешь сам.

Что сейчас умеет и не умеет upgrade()
Функция upgrade() в index.ts:66-80 делает ровно одно: итерирует целевую схему и создаёт отсутствующие stores + индексы. Это декларативное «добавляющее» обновление, и у него три дыры:

1. Нет удаления (stores и индексов)
db.createObjectStore есть, но db.deleteObjectStore(name) и store.deleteIndex(name) не вызываются нигде. Схема может только расти. Если завтра понадобится переименовать store или удалить устаревший индекс — это невозможно.

2. Нет цепочки миграций
upgrade() не знает, откуда идёт переход. Функция получает только db, а не полный IDBVersionChangeEvent — значит недоступны event.oldVersion, event.newVersion, event.target.transaction. Переход 1 → 5 ничем не отличается от 4 → 5. В реальности у них разный набор шагов, и пропустить промежуточные нельзя: v2 мог добавить поле, которое v3 переименовал, — если ты встретишь пользователя, открывшего приложение впервые за полгода, его БД на v1 должна пройти все шаги v2, v3, v4.

3. Нет версионной транзакции для данных
onupgradeneeded — единственное место, где одновременно можно (а) менять схему и (б) преобразовывать данные. Эта спец-транзакция (event.target.transaction) имеет режим versionchange, охватывает все stores, и только она умеет создавать/удалять stores. Сейчас обёртка её в callback не пробрасывает — значит backfill типа «проставить folderId: null всем нотам» в принципе негде написать. Если попытаться сделать это в отдельной транзакции после onsuccess, ты словишь race: другой tab с v1 может успеть записать в БД до твоего backfill'а.

Темы, которые нужно изучить (не код, а концепции)
Перечислено в порядке приоритета. Для каждой темы — что именно понять, а не только прочитать.

A. IDBVersionChangeEvent и versionchange-транзакция
MDN страницы: IDBVersionChangeEvent, IDBTransaction (режим versionchange).

Что именно в event.oldVersion если БД создаётся впервые (подсказка: 0).
Почему event.target.transaction — единственная транзакция, в которой разрешены schema-изменения.
Почему она автоматически коммитится, когда вернёшь управление из обработчика — и что произойдёт, если внутри неё сделать await до неявного yield в event loop.
В каком порядке запускаются onupgradeneeded → onsuccess.
B. Удаление схемы
IDBDatabase.deleteObjectStore(name)
IDBObjectStore.deleteIndex(name) (нужно сначала достать store из versionchange-транзакции через tx.objectStore(name))
Интроспекция: db.objectStoreNames (DOMStringList), store.indexNames. Понадобится, чтобы написать идемпотентные миграции («удали, если есть»).
C. Курсоры для data-миграций
IDBObjectStore.openCursor() внутри versionchange-транзакции.
cursor.update(newValue) для in-place изменения записи.
Почему это называют «upgrade-cursor» и почему он должен работать в той же транзакции, что и schema-change — иначе конкурентная запись с другой вкладки ломает инвариант.
D. Блокировки между вкладками при upgrade
Это понадобится в полной мере в Phase 3, но проигнорировать нельзя даже сейчас:

onblocked на IDBOpenDBRequest — когда старая вкладка держит v1, а новая хочет v2.
onversionchange на самом IDBDatabase — приходит в старую вкладку, и она обязана закрыть соединение, иначе новая зависнет.
Текущая обёртка ловит onblocked как ошибку и onversionchange не слушает вообще. Это ок для single-tab демо, но не для PWA.
E. Тестирование миграций
fake-indexeddb — in-memory polyfill, позволяющий прогонять миграции в Node/vitest без браузера.
Идея теста: создать БД на v1, записать фикстурные данные, закрыть, открыть на v2, проверить что данные мигрировали и индексы появились.
Это единственный способ реально проверить цепочку 1→2→3, а не только «свежий пользователь получает v3».
F. (Опционально для общей картины) Как это делает idb от Jake Archibald
Не для копирования, а чтобы увидеть устоявшиеся идиомы: в idb.openDB(name, version, { upgrade(db, oldVersion, newVersion, tx, event) { ... } }) callback получает всё сразу. Это минимально-необходимый API — любая твоя абстракция сверху должна хотя бы его обеспечивать.

Архитектурные развилки
Три направления, у каждого свои trade-offs. Выбор не очевиден и зависит от того, чему ты хочешь научиться.

Вариант 1: Declarative diff (продолжение текущего подхода)
Сравниваешь целевую схему с db.objectStoreNames / store.indexNames и генерируешь список операций (create/delete для stores и индексов).

Плюсы: одна целевая схема как source of truth; типы выводятся из неё; меньше копипасты. Минусы:

Data-миграции (backfill, преобразование полей, split/merge записей) не выражаются декларативно. Для них всё равно нужен escape hatch — и ты приходишь к гибриду.
Переименование поля неотличимо от «удалить старое + добавить новое» → потеря данных.
Не показывает сам процесс миграции, а это как раз то, что стоит пощупать руками в учебном проекте.
Вариант 2: Imperative migration chain (Dexie/Rails-style)
Массив миграций, индексированный по целевой версии:

migrations[2] = (ctx) => { /*что делаем при переходе на v2 */ }
migrations[3] = (ctx) => { /* что делаем при переходе на v3*/ }
В onupgradeneeded цикл: для каждого v от oldVersion + 1 до newVersion вызвать migrations[v](ctx). Внутри ctx — db, versionchange-tx, хелперы createStore / deleteStore / addIndex / deleteIndex / forEachRecord (курсор-хелпер).

Плюсы:

Явная цепочка: 1→5 проходит через 2, 3, 4 — никакой магии.
Data-миграции выражаются ровно там же, где schema-миграции, в одной транзакции.
Ближе всего к тому, что ты сам описал как желаемое («цепочка +1»).
Максимум контакта с сырыми IDB API — матчит цель проекта.
Минусы:

Нет единой target-схемы — приходится «читать миграции», чтобы понять актуальное состояние.
Типы для stores придётся объявлять отдельно (не выводятся из миграций).
Легко забыть обновить какую-то миграцию и получить расхождение «то, что получает свежий пользователь» vs «то, что получает давний пользователь».
Вариант 3: Гибрид (рекомендую)
Целевая схема декларативно — но только для типизации (какие stores, какие записи, какие индексы есть сейчас). Инфраструктура её не применяет к БД.
Миграции императивные, цепочкой (вариант 2).
Тест-инвариант: прогон всех миграций с v0 на свежей БД даёт состояние, эквивалентное целевой схеме. Это покрывается тестами на fake-indexeddb — «свежий пользователь» и «пользователь с v1» получают одно и то же.
Это архитектура, к которой пришли почти все зрелые ORM'ы поверх IDB. Для учебного проекта даёт максимум понимания при разумной сложности.

Рекомендуемое направление работы
Разбить на четыре независимо-завершаемых этапа. Каждый даёт работающее приложение.

Этап 1. Рефакторинг upgrade() под полный event
Цель — просто пробросить IDBVersionChangeEvent внутрь. Ещё без цепочки, без удалений.

upgrade получает (db, oldVersion, newVersion, tx).
Текущая логика «добавить отсутствующее» остаётся, но теперь живёт в контексте, где видно oldVersion.
Это чисто механическое изменение — проверка, что ты понял, как устроен event.
Этап 2. Миграции как цепочка
Ввести migrations: Record<number, MigrationFn> в DataBaseOptions вместо (или рядом с) schemas.

Написать runner, который в onupgradeneeded итерирует от oldVersion + 1 до newVersion.
Дать миграции контекст: { db, tx, createStore, deleteStore, addIndex, deleteIndex, forEachRecord }.
Хелперы — тонкие обёртки вокруг raw API, без магии.
Этап 3. Первая реальная миграция v1 → v2 для Folders
Это уже работа Phase 2.1, но она — лакмус для этапов 1–2. Если цепочка спроектирована правильно, миграция должна:

Создать store folders с keyPath id и индексом parentId.
Создать индекс folderId на store notes.
Пройти курсором по всем существующим нотам и проставить folderId: null.
Всё — в одной versionchange-транзакции. Если любой шаг упадёт, БД остаётся на v1 целиком.
Если в этот момент захочется написать миграцию вне upgrade-функции (например, «сначала создам store, потом в отдельной транзакции залью данные») — это сигнал, что цепочка спроектирована неправильно.

Этап 4. Блокировки между вкладками
Обработать onblocked (сейчас возвращает ошибку — это ок как заглушка) и onversionchange на открытом IDBDatabase (сейчас не обрабатывается вовсе).

Минимум: на versionchange закрывать соединение и возможно показывать юзеру «обнови страницу».
Полное решение — координация через BroadcastChannel, но это явно Phase 3 (уже заложено в PHASE2.md).
Вопросы, которые стоит себе задать по пути
Эти вопросы полезнее любой готовой схемы — они проверяют, что ты понял модель, а не просто скопировал API.

Что произойдёт, если внутри миграции ты сделаешь await fetch(...) — закоммитится ли versionchange-транзакция до того, как ты к ней вернёшься? Почему?
Если миграция v2 добавила индекс folderId, а миграция v3 его удалила — что должен получить пользователь, открывший приложение впервые на v3? (Подсказка: не «пустой результат v2 шага».)
Как протестировать, что цепочка v0 → v3 эквивалентна цепочке v1 (с данными) → v3? Какой фикстурный набор данных нужен?
Что делает браузер, если onupgradeneeded выбросил исключение на половине — остаётся ли БД в v1 или «порчено наполовину»?
Как ты узнаешь в проде, что у кого-то из пользователей миграция провалилась? (Ответ: текущая обёртка возвращает Result<IDBError> — но кто его читает в vaultDB singleton?)
Что осознанно вынесено за скоп
BroadcastChannel-координация между вкладками при upgrade — Phase 3, отдельная задача. Сейчас достаточно корректно обработать onversionchange.
Rollback миграций — IndexedDB не даёт downgrade по версии в принципе (версия только растёт). Если миграция сломалась в бою — это ручной incident, а не «откатись на v1».
Схема как source of truth для рантайма — если пойдёшь по гибриду, целевая схема нужна только для типов; её не надо применять к БД.
Репозиторный слой (IDBNoteRepository, IDBFolderRepository) — он неизбежно появится, но это ортогонально миграциям. Сначала разбираешься с эволюцией схемы, потом заворачиваешь вызовы vaultDB.* в репозитории.

Verification
Минимальная проверка, что новая архитектура работает:

Свежий пользователь открывает приложение на v2 → получает stores notes + folders и индекс folderId.
Пользователь с v1 (несколько заметок) открывает приложение на v2 → видит те же заметки, у всех folderId === null, появилась возможность создавать папки.
Прогон миграций на fake-indexeddb в vitest: фикстура с v1-данными → await openDB(...v2) → проверка состояния схемы и данных.
Ручная проверка onversionchange: открыть приложение в двух вкладках, задеплоить новую версию с v3 миграцией, открыть третью — старые не должны намертво блокировать обновление.
